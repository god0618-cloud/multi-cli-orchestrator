把多个 AI Coding CLI 组织成可监管、可审计、可回放的小团队

Multi-CLI Orchestrator 不替代 Codex、Claude Code、Kimi Code、Mimo Code 或 CodeWhale。它把这些 CLI 当成“工位”，用任务状态、workflow gate、artifact evidence、run ledger 和 boss dashboard 让复杂 AI 协作真正工程化。

1. 为什么需要它

很多开发者已经不是只用一个 AI 工具。Codex 适合本地工程执行，Claude Code 适合长上下文规划，Kimi Code 适合前端快速落地，Mimo Code 适合调研弹药，CodeWhale/DeepSeek 适合红队审查。问题不在于这些工具不强，而在于复杂任务里缺少一个共同控制面。

协作痛点没有控制面时的表现项目提供的解法
状态分散每个 CLI 窗口都有一部分上下文任务工作区保存共同事实
交接靠人用户复制 prompt 和文件路径dispatch queue / claim / complete
门禁靠自觉prompt 写“自检通过再继续”workflow gate 机器判定
完成不可验回答说完成，但缺证据链artifact / ledger / replay
自动化不可见不知道为什么继续或停下boss dashboard + observe recommendation
项目的核心判断：多 CLI 协作不是多开几个窗口，而是要有一个外部、可审计、可停止的控制面。

2. 核心概念

CLI Workstation

外部 AI Coding CLI。它可以接收任务、输出产物，但必须被 adapter、sandbox、timeout、quota 和 evidence gate 管住。

Task Workspace

每个任务的本地事实空间，包含 LOOP_SPEC.json、RUN_LEDGER.json、plan.json、dispatch 和 artifacts。

Workflow Gate

阶段推进条件，例如文件存在、artifact 已登记、dispatch 全部结束、无失败、需要用户决策等。

Sandbox Contract

定义 worker 的读写范围、凭据边界、端口、网络、执行权限和产物要求。

Run Ledger

可回放运行账本。记录任务、dispatch、决策、证据、失败和最终 verdict。

Boss Dashboard

面向人的控制台，回答谁在干活、什么阻塞、是否需要拍板、证据在哪里。

3. 系统架构

User / Boss -> mco CLI -> Task Lifecycle Core -> LOOP_SPEC.json / SANDBOX_CONTRACT.json -> Workflow Gates -> Dispatch Queue -> CLI Adapters -> Evidence Artifacts -> RUN_LEDGER.json -> Boss Dashboard -> User / Boss
分层职责
CLI 层mco 命令入口、任务创建、状态查询、workflow 操作。
Task Core创建任务、登记事件、管理 artifact、读取任务事实。
Workflow Core计算 phase、评估 gate、给出下一步建议。
Dispatch Corequeue、claim、complete、wave、execute。
Adapter Layer描述和执行不同 CLI 的能力边界。
Evidence Layer保存报告、日志、截图、usage snapshot、执行结果。
Dashboard可视化状态、门禁、证据、风险和升级点。

5. 严格门禁自闭环

v5.0 的关键突破是 workflow observe 和 workflow loop。系统不是无限自动跑，而是在每一步读取任务证据，给出 advance、wait、escalate 或 complete。

plan
execute
verify
close
replay
推荐动作含义
advance当前阶段 gate 已通过，可以推进。
wait证据不足、dispatch 未完成或条件未满足。
escalate出现失败、阻塞或用户决策门。
complete工作流已经闭环完成。
 
mco orchestrate-start "Strict product task" \
  --template strict-self-closing \
  --workspace .mco-workspace

mco workflow observe <task_id> --workspace .mco-workspace
mco workflow loop <task_id> --workspace .mco-workspace --max-steps 1

6. 安全边界

边界v5.0 策略
任意 shell不支持无约束任意 shell。
generic execution只允许窄范围安全命令。
Claude Codeclaude --print、禁用 tools、禁用 session persistence、timeout、output limit、budget cap。
Kimi Codekimi --prompt、timeout、output limit、transcript artifact。
manual-only CLI未通过 readiness 的 CLI 不进入自动派发链路。
workflow loop必须有 --max-steps，遇到缺证据、失败、阻塞、用户决策门即停。
自动化不是越多越好，而是证明过边界的自动化才进入闭环。

10. 路线图

阶段方向
近期增强更多 workflow template、更清晰 user decision gate、更强 dashboard 摘要、adapter smoke 结果对比。
中期探索更成熟的非交互式 CLI adapter 生命周期、quota/cost/cancel/timeout 策略、并发 worker 冲突治理。
长期愿景从本地 CLI 控制面走向可复用 Agent OS 基础设施，多端控制台和组织级治理策略。方向不等于锁死的版本承诺。

11. 项目定位总结

Multi-CLI Orchestrator 的价值不是“让 AI 更像人”，而是“让 AI CLI 的协作更像工程”。

目标 -> 计划 -> 派发 -> 执行 -> 证据 -> 验证 -> 门禁 -> 回放 -> 复盘

它不接管你的 CLI，不污染你的记忆，不承诺魔法。它只把协作、门禁、证据和停止条件变成工程对象。这就是 v5.0 的核心：严格门禁下的多 CLI 自闭环。
