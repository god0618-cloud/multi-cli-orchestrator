01 / Pain

同时使用多个 AI CLI 后，真正痛的不是模型不够聪明

过去一段时间，我同时在用很多 AI Coding CLI：Codex、Claude Code、Kimi Code、Mimo Code、CodeWhale、DeepSeek 相关 CLI。

它们各有长处。有的更适合架构判断，有的前端审美更好，有的适合搜集资料，有的适合做红队审查，有的在本地工程执行上更稳。

但真正做复杂项目时，我发现最大的问题不是“哪个模型更聪明”，而是协作过程会变成一团散落的上下文。

任务状态散在多个聊天窗口里
上下文靠人工复制粘贴
谁做了什么、失败在哪里，很难回放
每个 CLI 的记忆和成长机制可能互相污染
自动化之后容易无限跑、乱消耗额度
未准备好的 CLI 可能被误派任务

所以我做了 Multi-CLI Orchestrator。它不是一个新的 AI Coding CLI，而是一个控制平面。

02 / Idea

CLI 是工位，不是子进程

很多 Agent 框架强调“一个主 Agent 调多个 subagent”。这个思路很有价值，但它通常仍然处在同一个 CLI、同一个模型供应商、同一个运行时边界里。

我想要的是另一种结构：让不同 CLI、不同模型、不同工具生态，作为真正独立的工位参与复杂任务。

控制平面负责组织协作；具体干活的 CLI 保留自己的模型、认证、上下文习惯和工具生态。

散落式多 CLI

  • 人工复制 prompt
  • 任务状态在聊天记录里
  • 失败原因难回放
  • 靠用户记住下一步

受监管的 CLI 工位

  • dispatch 明确派棒
  • task workspace 保存状态
  • RUN_LEDGER 可回放
  • dashboard 显示阻断和证据
03 / Design

先把协作变得诚实、可见、可审计，再谈更强自动化

Multi-CLI Orchestrator 的设计顺序很保守：先控制平面，再自动化。

1Workspace

创建本地任务工作区

2Loop

用 LOOP_SPEC 定义阶段和门禁

3Dispatch

把任务派给具体 CLI 工位

4Adapter Gate

检查工位是否真的准备好

5Artifact

每个动作留下证据

6Replay

任务结束后可复盘

组件它解决什么问题
task workspace任务状态不再依赖聊天上下文。
RUN_LEDGER.json记录任务推进过程，支持 replay。
dispatch/agent-inbox每个 CLI 工位都有明确收件箱。
adapter matrix显性化哪个 CLI 真正可用。
sandbox contract定义工位的读写范围和凭证策略。
dashboard.html给人看的老板视角。
04 / v4.0

v4.0：有界多 worker dispatch wave

v4.0 新增了一个关键能力：一次派发多个 worker，但仍然保持门禁和证据。

mco dispatch wave <task_id> --spec wave.json --require-ready --workspace .mco-workspace

wave.json 示例：

{
  "title": "Sprint review wave",
  "workers": [
    {
      "agent": "generic-cli",
      "title": "API drift review",
      "instructions": "Check API contract drift and report evidence."
    },
    {
      "agent": "kimi-code",
      "title": "Frontend polish review",
      "instructions": "Review mobile UI fit and report findings."
    }
  ]
}
最多 6 个 workers

防止 wave 变成不可控的自动化散射。

逐个 adapter gate

未达到 READY_SUPERVISED 的工位不会收到任务。

阻断也写证据

blocked dispatch 会记录原因，便于复盘和修复。

重要边界：v4.0 不是无约束真实并发执行。它先完成可监管的多工位派棒，因为多 Agent 系统真正难的不是跑起来，而是跑起来之后还能看得懂、停得住、审得清。
05 / Try

最小体验命令

git clone https://github.com/god0618-cloud/multi-cli-orchestrator.git
cd multi-cli-orchestrator

python -m venv .venv
source .venv/bin/activate
pip install -e .

mco init --workspace .mco-workspace
mco doctor --workspace .mco-workspace
mco demo walkthrough --workspace .mco-walkthrough --output-dir .mco-walkthrough-output

想体验 v4.0 的 dispatch wave：

mco task create "Demo multi-worker wave" --json --workspace .mco-workspace
mco dispatch wave <task_id> --spec wave.json --require-ready --workspace .mco-workspace

准备好的工位会进入队列；没准备好的工位会被阻断并留下证据。

06 / Safety

这个项目默认保守

不默认污染记忆

不默认写稳定知识库，不默认改 CLI 原生 memory / profile。

不默认任意执行

Generic command 只有严格 allowlist；真实 provider smoke 必须 opt-in。

不默认启用新工位

新 adapter disabled-by-default，必须靠证据晋升。

能力当前状态
本地 workspace / task / ledger / artifact已实现
dashboard / replay已实现，静态 HTML + 文本/HTML replay
Claude Code / Kimi Code adapter已实现，受限执行
dispatch wavev4.0 已实现
Mimo / CodeWhale模板或待接入，不默认启用
真实并发 provider 执行尚未开放
