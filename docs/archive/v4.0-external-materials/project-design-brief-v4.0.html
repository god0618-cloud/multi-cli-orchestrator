Multi-CLI Orchestrator 项目设计说明

一个本地优先的多 CLI 协作控制平面，把 Codex、Claude Code、Kimi Code 等 AI Coding CLI 组织成可监管、可审计、可回放的“工位”。

1. 项目定位

一句话：Multi-CLI Orchestrator 让多个 AI Coding CLI 像一个受监管的小团队一样工作：任务可以被拆分、派发、执行、审计、回放，而不是散落在多个聊天窗口里。
控制平面

任务状态、运行账本、派棒、门禁、artifact 和 dashboard 不绑定任一模型会话。

CLI 工位

每个 CLI 保留自己的模型、认证、上下文习惯和工具生态，以 adapter 方式受控接入。

证据闭环

先记录事实和产物，再决定是否沉淀知识，避免直接污染长期记忆。

2. 为什么不是再做一个 Agent 框架

问题传统做法MCO 的设计
不同模型参与常在同一 CLI 的 subagent 体系内把不同 CLI 当作独立工位，保留差异
任务状态依赖聊天上下文或复制粘贴任务目录、ledger、artifact 本地持久化
自动化边界容易进入无限跑或隐式执行循环、worker 数、门禁、执行命令全部有界
能力接入容易先宣称可用disabled-by-default，证据通过后晋升

4. v4.0 能力

单 worker dispatch 
mco dispatch queue <task_id> \
  --agent kimi-code \
  --title "Frontend pass" \
  --instructions "Polish mobile UI" \
  --require-ready
多 worker dispatch wave 
mco dispatch wave <task_id> \
  --spec wave.json \
  --require-ready

Wave Spec

{
  "title": "Sprint review wave",
  "workers": [
    {
      "agent": "generic-cli",
      "title": "API drift review",
      "instructions": "Check API contract drift and report evidence."
    },
    {
      "agent": "kimi-code",
      "title": "Frontend polish review",
      "instructions": "Review mobile UI fit and report findings."
    }
  ]
}
v4.0 边界：dispatch wave 是有监督多工位派棒，不是无约束真实并发执行。一个 wave 最多 6 个 workers；每个 worker 仍然经过 adapter gate；不达标的工位只会留下 blocked evidence，不会收到 inbox 文件。

5. 演示主线

步骤命令 / 画面讲解重点
1打开 GitHub README这是多 CLI 协作控制平面，不是单一 Agent 框架。
2mco init / mco doctor本地优先，先建立 workspace 和治理入口。
3mco task create --json复杂任务有独立目录和账本，不靠聊天上下文续命。
4mco adapter matrix --doctor --html显性化每个 CLI 是否真的可用，避免“什么都能做”的幻觉。
5mco dispatch wave --require-ready多工位派棒，但每个工位都受 readiness gate 约束。
6mco dashboard / mco run replay --html老板视角可见，任务过程可回放。

6. 安全与治理设计

不默认污染记忆

不默认写稳定知识库，不默认改 CLI 原生 memory / soul / profile。

不默认任意执行

Generic command 只有严格 allowlist；真实 provider smoke 明确 opt-in。

不默认启用新工位

新 adapter disabled-by-default，必须经过 manifest、sandbox、fixture、contract test、smoke evidence。

8. 发文宣传角度

推荐标题

我把多个 AI Coding CLI 组织成了一个可审计的小团队

不再复制粘贴 prompt：一个本地优先的 Multi-CLI Agent OS

为什么我不满足于 subagent，而要做 Multi-CLI Orchestrator

推荐开场

我最近一直在用多个 AI Coding CLI：Codex、Claude Code、Kimi Code、Mimo、CodeWhale。它们各有长处，但真正做复杂项目时，最大的问题不是模型够不够聪明，而是协作状态不可见、任务交接靠复制粘贴、证据散在不同会话里。

10. 收束话术

Multi-CLI Orchestrator 的目标不是制造一个“无所不能的 AI 大脑”，而是把多个已经存在的 AI CLI 组织成一个可监管、可审计、可回放的小团队。它把任务状态从聊天窗口里抽出来，把能力接入做成 adapter，把安全边界写成 sandbox，把协作过程写进 ledger，把结果沉淀为 artifact。v4.0 开始，它已经支持有界多 worker dispatch wave，这意味着多 CLI 协作从“人工复制粘贴”进入了“控制平面派棒”的阶段。
